WHAT THIS DESIGN SOLVES
================================================================================

PROBLEM (Current State):
  • Video workflows lack job-level persistence
  • Worker crash at Stage 3 → entire pipeline restarts from Stage 1
  • 2+ hours GPU compute LOST per failure
  • No recovery mechanism exists
  • Error handling scattered across codebase

SOLUTION (Proposed):
  • BullMQ (RQ) job queue with Redis backend
  • PostgreSQL checkpoints saved after each stage
  • Automatic resume from last successful checkpoint
  • 3-tier error handling (E4 taxonomy):
    - Immediate retry (60s backoff) for transient errors
    - Delayed retry (300s backoff) for recoverable errors
    - Manual escalation for fatal errors
  • Supabase Storage for intermediate artifacts (with TTL)
  • Structured logging for cost tracking

IMPACT:
  • Job success rate > 99% (with retry)
  • GPU time saved: 2+ hours per crash (resume vs restart)
  • Error visibility: 100% (all errors logged to PostgreSQL)
  • Cost tracking: 30-50% reduction (with Gap 3 integration)
  • User experience: Transparent recovery (no manual intervention)

================================================================================
ARCHITECTURE COMPONENTS
================================================================================

Component 1: Job Queue
  Technology: RQ (Python) + Redis
  Purpose: Distribute work, persist job state
  Dependencies: redis, rq, rq-scheduler

Component 2: Job Classes
  Count: 5 (one per video stage)
  Base: BaseVideoJob (abstract)
  Subclasses:
    - Stage1ScriptGenerationJob (LLM)
    - Stage2AudioSynthesisJob (TTS)
    - Stage3VoiceSegmentationJob (VAD)
    - Stage4VideoMattingJob (GPU)
    - Stage5RenderingJob (Compose)

Component 3: Checkpoints
  Database: PostgreSQL (autoflow_checkpoints table)
  Schema: workflow_id, stage, attempt, status, outputs (JSONB), execution_time_ms
  Purpose: Persist stage outputs for resume on crash
  Indices: workflow_id, stage+status

Component 4: Error Handling
  Database: PostgreSQL (autoflow_errors table)
  Taxonomy: E4_TIMEOUT, E4_GPU_UNAVAIL, E4_API_QUOTA (retryable)
            E4_OOM, E4_AUTH_FAIL (fatal)
  Strategy: Immediate retry → Delayed retry → Escalate → Manual

Component 5: Artifact Storage
  Provider: Supabase Storage (R2-compatible)
  Path: autoflow-artifacts/{workflow_id}/stage{N}_{key}
  TTL: 24h (intermediate), 7d (matte), 90d (final video)
  Purpose: Persistent storage for intermediate outputs

Component 6: Worker
  Type: RQ Worker (processes jobs from Redis queue)
  Scaling: 1+ instances (horizontal)
  Logging: JSON structured logs to PostgreSQL
  Monitoring: Queue depth, worker CPU, job success rate

Component 7: API
  Endpoint: POST /api/videos/submit
  Endpoint: GET /api/videos/{workflow_id}/status
  Purpose: Enqueue workflows, poll status

Component 8: Metrics
  Database: PostgreSQL (autoflow_metrics table)
  Tracked: execution_time_ms, tokens_used, gpu_seconds, cost_usd
  Purpose: Cost tracking, performance analysis
