================================================================================
ARCHITECTURE AT A GLANCE
================================================================================

5-STAGE VIDEO PIPELINE:
  Stage 1: Script Generation    (LLM)          → script.json
  Stage 2: Audio Synthesis      (TTS)          → audio.wav
  Stage 3: Voice Segmentation   (VAD)          → segments.json
  Stage 4: Video Matting        (GPU)          → matte.mp4
  Stage 5: Final Rendering      (Compose)      → final_video.mp4

PERSISTENCE LAYERS:
  Redis Queue         ← Job queue (RQ library)
  PostgreSQL          ← Checkpoints (JSONB), errors (E4), metrics (cost)
  Supabase Storage    ← Artifacts with TTL (24h/7d/90d)

ERROR HANDLING (E4 TAXONOMY):
  TRANSIENT           → Retry 60s, 3x   (timeout, rate limit)
  RECOVERABLE         → Retry 300s, 2x  (GPU unavail, API quota)
  FATAL               → Escalate, 0x    (OOM, auth fail)

CRASH RECOVERY:
  Stage 3 crashes → RQ detects timeout → Load Checkpoint 2
                 → Re-enqueue Stage 3 with backoff
                 → Continue to Stages 4-5 normally

================================================================================
WHAT THIS SOLVES
================================================================================

PROBLEM (Current State):
  • No job-level persistence
  • Crash at Stage 3 = entire pipeline restarts from Stage 1
  • 2+ hours GPU compute WASTED per failure
  • No recovery mechanism
  • Error handling scattered across codebase

SOLUTION (Designed):
  • RQ job queue with Redis backend
  • PostgreSQL checkpoints after each stage
  • Automatic resume from last successful checkpoint
  • Structured error handling (E4 taxonomy)
  • Supabase artifact storage with TTL
  • Comprehensive logging for debugging

IMPACT:
  • Job success rate > 99% (with retry)
  • GPU time saved: 2+ hours per crash (resume vs restart)
  • Error visibility: 100% (all logged to PostgreSQL)
  • Cost tracking: Enable 30-50% reduction (with Gap 3)
  • User experience: Transparent recovery (no manual intervention)

TECHNICAL SPECIFICATION
================================================================================

COMPONENTS:
  • Job Queue Library:      RQ 1.14.0 (Python)
  • Redis Backend:          6.0+ (existing on VPS)
  • Checkpoints:            PostgreSQL (new tables)
  • Error Logging:          PostgreSQL (new tables)
  • Metrics:                PostgreSQL (new tables)
  • Artifact Storage:       Supabase Storage (existing, new bucket)

DATABASE SCHEMA:
  autoflow_checkpoints      ← workflow_id, stage, attempt, outputs (JSONB)
  autoflow_errors           ← workflow_id, stage, error_code, traceback
  autoflow_metrics          ← workflow_id, stage, execution_time, cost
  autoflow_error_taxonomy   ← code, category, is_retryable, retry_delay

JOB CLASSES (5 total):
  BaseVideoJob              ← Abstract base with checkpoint/execute
  Stage1ScriptGenerationJob ← LLM input: brief → output: script
  Stage2AudioSynthesisJob   ← TTS input: script → output: audio
  Stage3VoiceSegmentationJob ← VAD input: audio → output: segments
  Stage4VideoMattingJob     ← GPU input: script+segments → output: matte
  Stage5RenderingJob        ← Compose input: matte+audio → output: video

API ENDPOINTS (2 total):
  POST /api/videos/submit           ← Enqueue workflow (returns workflow_id)
  GET /api/videos/{workflow_id}/status ← Poll status + checkpoint data

WORKER:
  RQ Worker process listening on Redis queue 'autoflow-jobs'
  Processes jobs sequentially (horizontal scaling available)
  Hooks: job success → log metrics, job failure → resume_from_checkpoint()
