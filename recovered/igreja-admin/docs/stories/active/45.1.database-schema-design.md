### Code Examples

**Sample Migration (001_create_communities.sql):**
```sql
CREATE TABLE IF NOT EXISTS communities (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  church_id UUID NOT NULL REFERENCES churches(id) ON DELETE CASCADE,
  name VARCHAR(255) NOT NULL,
  description TEXT,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE INDEX idx_communities_church_id ON communities(church_id);
CREATE INDEX idx_communities_created_at ON communities(created_at DESC);
```

**Schema Validation Test:**
```typescript
describe('Database Schema', () => {

    const { data } = await supabase

    })
    expect(data).toContainEqual(
      expect.objectContaining({
        column_name: 'id',
        data_type: 'uuid'
      })
    )
  })
})
```

## 📝 Implementation Notes

### Key Decisions
1. **UUID Primary Keys:** Use gen_random_uuid() for all IDs (consistency with existing schema)
2. **Timestamps:** All tables have created_at/updated_at for audit trail
3. **Foreign Keys:** Cascade deletes where appropriate (e.g., community → threads)
4. **Indexing Strategy:**
   - Foreign keys: Indexed for join performance
   - Date columns: Indexed DESC for "recent first" queries
   - Status columns: Indexed for filtering
5. **RLS Preparation:** Document policies (will be implemented in Story 45.3)

### Assumptions
- Existing `churches` table available (from Story 43.4)
- Existing `donors` table available (from Story 43.4)
- All users tracked via NextAuth session/JWT
- Standard PostgreSQL extensions available

### Questions for @po
- Should deleted communities/projects be soft-deleted (status field) or hard-deleted?
- Should member profiles be created automatically or manually?
- What retention policy for admin_logs (archive after N months)?
