# Story 4.1: Opt-in Usage Metrics Collection

## Objective

Implement anonymous, opt-in telemetry for the AIOX CLI. Tracks key success metrics — time-to-first-value, onboarding completion rate, command discovery rate — stored locally as JSON with no external network calls. Users have full control via CLI commands.

## Agent

@dev (Dex)

## Status

Draft

## Story

**As a** developer adopting AIOX,
**I want** to opt in to anonymous usage metrics collection,
**so that** the AIOX team can measure onboarding effectiveness and improve the product based on real usage patterns — without any data leaving my machine unless I explicitly export it.

## Acceptance Criteria

- [ ] `aiox telemetry on` enables metrics collection and confirms with a message
- [ ] `aiox telemetry off` disables collection and confirms with a message
- [ ] `aiox telemetry status` displays current enabled/disabled state and local metrics summary
- [ ] `aiox telemetry export` outputs the full local metrics JSON to stdout for manual sharing
- [ ] Metrics are stored locally in `.aiox/metrics/` as JSON files — no network calls by default
- [ ] Tracks: time-to-first-value (time from install to first successful `aiox` command), onboarding completion rate (% of quickstart steps completed), command discovery rate (unique commands used vs total available)
- [ ] Opt-in state is persisted in `.aiox/telemetry.json`
- [ ] When disabled (default), zero instrumentation runs — no performance impact
- [ ] Zero external dependencies added to the project
- [ ] Unit tests cover: enable/disable toggle, metric recording, export output, status display

## Dev Notes

**Architecture Constraint — CLI First (Article I):**
All telemetry functionality MUST be accessible and complete via CLI alone. No UI dependency.

**Zero External Dependencies:**
Do not add npm packages. Use Node.js built-ins: `fs`, `path`, `crypto` (for session IDs), `os`.

**File Locations:**
- Telemetry state: `.aiox/telemetry.json`
- Metrics data: `.aiox/metrics/YYYY-MM-DD.json`
- Module source: `bin/` or the existing CLI entry point directory (follow current CLI structure)
- Tests: `tests/` matching existing test file patterns

**Atomic File Writes Pattern:**
```javascript
// Write to temp file first, then rename
fs.writeFileSync(tmpPath, JSON.stringify(data, null, 2));
fs.renameSync(tmpPath, targetPath);
```

**Metric Definitions:**
- `time_to_first_value`: milliseconds from install timestamp (stored at install) to first successful CLI invocation
- `onboarding_completion_rate`: count of completed quickstart steps / total quickstart steps
- `command_discovery_rate`: count of unique commands invoked / total commands registered in CLI

**Guard Pattern — Disabled State:**
```javascript
export function recordMetric(name, value) {
  const state = readTelemetryState();
  if (!state.enabled) return; // fast exit, zero overhead
  // ... write logic
}
```

**Testing:**
- Test framework: use whatever framework is already configured in `package.json`
- Test location: `tests/telemetry/` or existing test directory convention
- Mock `fs` calls in unit tests — do not write to actual `.aiox/` during tests
