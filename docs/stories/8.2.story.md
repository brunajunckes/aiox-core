# Story 8.2: Template Scaffold Generator

## Objective

Give developers a single CLI command to generate consistent, variable-interpolated scaffolds for the most common AIOX artefact types. Running `aiox scaffold component MyWidget` writes a ready-to-edit file in the correct location, eliminating copy-paste and manual boilerplate.

## Agent

@dev (Dex)

## Status

InReview

## Story

**As a** developer starting new components, modules, tests, stories, workflows, or squads,
**I want** to generate scaffold files from predefined templates via a single CLI command,
**so that** I can start new artefacts consistently without copying boilerplate or remembering directory conventions.

## Acceptance Criteria

- [x] `aiox scaffold <type> [name]` generates a scaffold from the corresponding template in `.aiox-core/data/scaffolds/`
- [x] `aiox scaffold list` prints all available scaffold types discovered from `.aiox-core/data/scaffolds/`
- [x] Supported types are: `component`, `module`, `test`, `story`, `workflow`, `squad`
- [x] Templates are stored in `.aiox-core/data/scaffolds/<type>.tmpl` (or `.tmpl.md` / `.tmpl.js` as appropriate)
- [x] Templates support variable interpolation for `{{name}}`, `{{date}}` (ISO date), and `{{author}}` (from git config `user.name`)
- [x] Generated files are written to the appropriate target directory per type (see Dev Notes for mapping)
- [x] If a file with the generated target path already exists, the CLI exits with code 1 and a clear message without overwriting
- [x] `aiox scaffold <type>` without a name interactively prompts for the name (stdin readline)
- [x] Zero external npm dependencies — implemented with Node.js built-ins only

## Dev Notes

**Architecture Constraint — CLI First (Article I):**
All scaffold logic is CLI-only. Templates live in `.aiox-core/data/scaffolds/` (L3, config-layer — read-only by developer commands, but writable by framework maintainers).

**Variable Interpolation:**
Use a simple regex replace: `/\{\{(\w+)\}\}/g`. Collect all tokens, resolve from vars map, throw with a list of unresolved tokens if any remain after substitution.

**Author Resolution:**
```js
const { execSync } = require('child_process');
let author = 'unknown';
try { author = execSync('git config user.name', { encoding: 'utf8' }).trim(); } catch {}
```

**Target Directory Creation:**
Use `fs.mkdirSync(dir, { recursive: true })` before writing the file so nested target paths are always valid.

[AUTO-DECISION] Template file extension → use `.tmpl.<ext>` (e.g. `.tmpl.js`, `.tmpl.md`) so editors apply syntax highlighting to the template source while `scaffold list` can still discover them by `.tmpl` substring. (reason: DX for template authors — no extension would mean no syntax highlighting)

**Relevant Source Tree:**
- CLI router: `bin/aiox.js`
- Existing command modules: `packages/cli/commands/`
- Scaffold templates: `.aiox-core/data/scaffolds/` (create directory if absent)
- Entity registry: `.aiox-core/data/entity-registry.yaml`

**Testing:**
- Test location: `tests/unit/scaffold/`
- Use `fs.mkdtemp` for isolated output directories
- Stub `execSync` to control author resolution in tests
- Test template engine and type-map independently from the command handler
