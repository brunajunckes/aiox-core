## Story
**As a** developer working with the AIOX CLI,
**I want** a command palette that lists all commands across all agents with fuzzy search filtering,
**so that** I can quickly discover and understand available commands without memorizing the full command inventory or switching to documentation.

## Acceptance Criteria

1. - [x] `aiox palette` (alias: `aiox commands`) command launches the command palette
2. - [x] Palette lists all commands across all agents in a unified view with columns: Command, Agent Owner, Description
3. - [x] Fuzzy search filter narrows the list as the user types — results update in real time
4. - [x] Selecting a command from the palette displays its usage example and the agent that owns it
5. - [x] Commands are sourced from agent definition files in `.aiox-core/development/agents/` — no hardcoded command lists
6. - [x] Palette is navigable with keyboard (arrow keys, enter to select, escape to exit)
7. - [x] When no input is given and user presses Enter on a command, show usage example and exit palette cleanly

## Dev Notes

**Source Tree — relevant paths:**
- Agent definition files: `.aiox-core/development/agents/` — same source used by Story 3.1
- CLI entry: `bin/aiox.js` — register `palette` and `commands` subcommands
- If Story 3.1 built a shared agent-file parser, reuse that module here (IDS principle: REUSE > ADAPT > CREATE)

**Fuzzy search approach:**
- Prefer a minimal, dependency-free implementation if the project has no existing fuzzy library
- If a fuzzy library is already in `package.json`, use it — do not add a duplicate
- Algorithm priority: prefix match first, then substring match, then character scatter

**Interactive terminal rendering:**
- Use the same interactive terminal library/approach already established in Story 2.1 (onboarding) and Story 3.1 (agent picker)
- Ensure `process.stdout.write` / readline state is restored on exit (both normal and Ctrl+C)
- Test on 80-column terminal width minimum

**Dependency on Story 3.1:**
- Story 3.1 builds the agent file reader and parser
- If Story 3.1 is complete, import its parser module
- If Story 3.1 is not yet merged, implement a local parser and note the duplication for later refactor

**CLI First principle (Constitution Article I):**
- This is a pure CLI feature — no UI dashboard component
- Must work fully in headless terminal environments (no GUI dependencies)
