# Story 8.4: Interactive CLI Dashboard

## Objective

Provide developers with a live terminal dashboard that surfaces system health, active story progress, recent command history, and test status in a single `aiox dashboard` command — built entirely with Node.js built-ins and ANSI escape codes, requiring no external TUI libraries.

## Agent

@dev (Dex)

## Status

InReview

## Story

**As a** developer working in a terminal environment,
**I want** a live-refreshing dashboard that shows system health, the active story, recent commands, and test status,
**so that** I have a persistent situational overview without switching between files and terminal windows.

## Acceptance Criteria

- [x] `aiox dashboard` launches a full-screen terminal dashboard that auto-refreshes every 5 seconds
- [x] The dashboard renders four sections: System Health, Active Story, Recent Commands, and Test Status
- [x] **System Health** section shows: CPU load (1-min average from `os.loadavg()`), memory used/total (from `os.totalmem`/`os.freemem`), and uptime in a human-readable format
- [x] **Active Story** section shows: active story ID (from `.aiox/active-profile.json` or current git branch), story title, and status line extracted from the story file
- [x] **Recent Commands** section shows the last 10 entries from `.aiox/command-history.jsonl` (each line: `{ ts, command }`)
- [x] **Test Status** section shows: last test run timestamp, pass/fail result, and count of passing/failing tests — sourced from `.aiox/test-last-result.json`
- [x] Pressing `Ctrl+C` exits the dashboard cleanly, restoring the original terminal state (cursor, alternate screen)
- [x] The dashboard uses ANSI escape codes for layout (alternate screen buffer, cursor positioning, color) with no external npm dependencies
- [x] All four sections are visible without scrolling on a standard 80x24 terminal
- [x] The refresh interval is configurable via `--interval <seconds>` flag (default: 5, min: 1, max: 60)

## Dev Notes

**Architecture Constraint — CLI First (Article I):**
Dashboard is a CLI observability surface — it reads data, never writes or controls. All data sources are files or `os` module; no network calls.

**ANSI Escape Sequences:**
```
Alternate screen on:  \x1b[?1049h
Alternate screen off: \x1b[?1049l
Hide cursor:          \x1b[?25l
Show cursor:          \x1b[?25h
Clear screen:         \x1b[2J
Move cursor:          \x1b[{row};{col}H
Reset color:          \x1b[0m
Bold:                 \x1b[1m
Green:                \x1b[32m
Red:                  \x1b[31m
Yellow:               \x1b[33m
Cyan:                 \x1b[36m
```

**Layout Budget (80x24 terminal):**
```
Row 1:    Header bar (title + timestamp)
Rows 2-7:  System Health (6 rows)
Rows 8-13: Active Story (6 rows)
Rows 14-19: Recent Commands (6 rows)
Rows 20-23: Test Status (4 rows)
Row 24:   Footer / refresh countdown
```

**Command History Format:**
`.aiox/command-history.jsonl` — one JSON object per line: `{ "ts": "<ISO>", "command": "<aiox ...>" }`.
The CLI writes to this file on each command invocation (future story). For Sprint 8, dashboard reads it — if absent, show empty placeholder.

**Test Last Result Format:**
`.aiox/test-last-result.json`:
```json
{ "ts": "<ISO>", "result": "pass", "passed": 42, "failed": 0 }
```

[AUTO-DECISION] Full clear-and-redraw vs partial update → full clear-and-redraw per tick is the baseline implementation for Sprint 8. Partial-update optimization is deferred to a future sprint. (reason: Significantly simpler implementation with negligible flicker at 5s interval; correctness over optimization for initial delivery)

**Relevant Source Tree:**
- CLI router: `bin/aiox.js`
- Dashboard modules: `packages/cli/dashboard/`
- Data files: `.aiox/command-history.jsonl`, `.aiox/test-last-result.json`, `.aiox/active-profile.json`
- Entity registry: `.aiox-core/data/entity-registry.yaml`

**Testing:**
- Test location: `tests/unit/dashboard/`
- Use `jest.spyOn(process.stdout, 'write')` to capture render output without printing to terminal
- Stub `os` module methods to return deterministic values
- Test collectors with fixture files in temp directories
- Do not test live `setInterval` loops — test the render and collect functions independently
