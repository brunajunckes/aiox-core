# Story 4.2: Onboarding A/B Testing Framework

## Objective

Lightweight A/B variant system for `aiox quickstart`. Assigns users deterministically to one of two variants (guided vs minimal), records which variant was run, and integrates with the Story 4.1 metrics system to measure which variant achieves higher onboarding completion.

## Agent

@dev (Dex)

## Status

Draft

## Story

**As a** developer working on AIOX product growth,
**I want** a simple A/B testing framework for the `aiox quickstart` flow,
**so that** we can empirically measure which onboarding variant (guided vs minimal) achieves higher completion rates without manual analysis.

## Acceptance Criteria

- [ ] 2 variants supported: `guided` (current full quickstart flow) and `minimal` (streamlined, fewer steps)
- [ ] Variant assignment is deterministic per installation — same user always gets same variant
- [ ] Variant assignment stored in `.aiox/experiment.json`
- [ ] `aiox experiment list` displays all active experiments and their variants
- [ ] `aiox experiment status` displays the current user's assigned variant and completion state
- [ ] `aiox quickstart` runs the variant assigned to the user
- [ ] Completion event is recorded via Story 4.1 telemetry (metric: `onboarding_completion_rate` tagged with variant name)
- [ ] Variant assignment uses a 50/50 random split on first run, then is locked
- [ ] If telemetry is disabled (Story 4.1), experiment tracking still records variant assignment locally but skips metric emission
- [ ] Unit tests cover: variant assignment determinism, experiment state persistence, variant routing in quickstart

## Dev Notes

**Dependency on Story 4.1:**
Import `recordMetric` from the telemetry module created in Story 4.1. Do not duplicate metric file writing logic. If Story 4.1 module is not present, this story is blocked.

**File Locations:**
- Experiment state: `.aiox/experiment.json`
- Module source: same directory as Story 4.1 telemetry module, consistent with CLI module pattern
- Tests: `tests/experiment/` or alongside existing test directories

**Variant Assignment Algorithm:**
```javascript
import { randomBytes } from 'crypto';

function assignVariant(variants) {
  const byte = randomBytes(1)[0];
  const index = byte % variants.length; // uniform for 2 variants
  return variants[index];
}
```

**Experiment Registry (hardcoded for now):**
```javascript
const EXPERIMENTS = [
  {
    id: 'quickstart-flow',
    variants: ['guided', 'minimal'],
    active: true,
  }
];
```

**Graceful Degradation:**
If `.aiox/experiment.json` is unreadable (corrupt JSON), catch the parse error, delete the file, and re-assign. Log a warning to stderr. Never crash the quickstart flow due to experiment state issues.

**Minimal Variant Steps:**
The `minimal` variant for quickstart is intentionally streamlined:
1. Verify AIOX is installed and configured
2. Run `aiox story create` to create their first story
3. Print success message with link to docs

**Testing:**
- Use the same test framework as Story 4.1
- Mock `crypto.randomBytes` to test determinism
- Test location: `tests/experiment/` or existing convention
