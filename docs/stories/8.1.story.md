# Story 8.1: Config Profiles & Environment Switching

## Objective

Give developers first-class support for managing multiple runtime environments through named configuration profiles. Running `aiox config profile use staging` instantly switches all CLI commands to read from the staging profile, eliminating manual edits to config files and reducing environment-switch errors.

## Agent

@dev (Dex)

## Status

InReview

## Story

**As a** developer working across dev, staging, and production environments,
**I want** to create, list, and switch named config profiles from the CLI,
**so that** I can safely isolate environment-specific settings and switch contexts without manually editing config files.

## Acceptance Criteria

- [x] `aiox config profile list` shows all available profiles stored in `.aiox/profiles/`, marking the currently active one
- [x] `aiox config profile use <name>` switches the active profile by writing the profile name to `.aiox/active-profile.json`
- [x] `aiox config profile create <name>` creates a new profile file at `.aiox/profiles/<name>.json` with a default schema; exits with an error if the profile already exists
- [x] Profiles are stored as individual JSON files at `.aiox/profiles/{name}.json`
- [x] The active profile is persisted in `.aiox/active-profile.json` as `{ "active": "<name>" }`
- [x] The CLI reads `.aiox/active-profile.json` at startup to determine the active profile; defaults to `dev` when the file is absent
- [x] `aiox config profile list` displays a table with columns: Name, Path, Active (boolean marker)
- [x] `aiox config profile use <name>` exits with a non-zero code and a clear message when the named profile does not exist
- [x] Zero external npm dependencies — implemented with Node.js built-ins only

## Dev Notes

**Architecture Constraint — CLI First (Article I):**
All profile logic is CLI-only. Profile files live in `.aiox/profiles/` (L4, always mutable).

**Default Profile Schema:**
```json
{
  "name": "<profile-name>",
  "createdAt": "<ISO timestamp>",
  "env": "dev",
  "settings": {}
}
```

**Active Profile File:**
```json
{ "active": "dev" }
```

**Startup Resolution:**
Read `.aiox/active-profile.json` once at CLI entry point. Store in a lightweight singleton. Commands that need env-aware behavior import the singleton rather than reading the file again.

**Relevant Source Tree:**
- CLI router: `bin/aiox.js` (subcommand dispatch)
- Existing config commands: `packages/cli/commands/`
- Profile storage: `.aiox/profiles/` (create if absent)
- Entity registry: `.aiox-core/data/entity-registry.yaml`

**Testing:**
- Test location: `tests/unit/config/`
- Use `fs.mkdtemp` to create isolated temp directories per test — never write to real `.aiox/`
- Test the store module independently from the command handlers (pure function tests)
