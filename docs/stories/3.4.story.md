## Story
**As a** developer at any point in the AIOX workflow,
**I want** a `aiox help` command that shows content relevant to my current state,
**so that** I receive actionable guidance matched to where I am in the process instead of a generic static help page.

## Acceptance Criteria

1. - [x] `aiox help` command detects current state from `.aiox/session-state.json` (produced by Story 3.3) and adapts its output accordingly
2. - [x] If no active story: output guides the user toward creating one ("No active story. Run `aiox agents` to discover agents, then `aiox sm` to create a story.")
3. - [x] If a story is active and status is InProgress: output shows relevant next actions for the current workflow phase with the specific story ID referenced
4. - [x] If an agent is active (from session state): output shows that agent's specific commands and scope
5. - [x] Fallback: if session state is missing or unreadable, `aiox help` shows standard static help (existing behavior is preserved — no regression)
6. - [x] `aiox help --raw` flag bypasses context detection and always shows the full static help page
7. - [x] Context detection completes in under 200ms (session state read is synchronous file I/O — no network calls)

## Dev Notes

**Depends on Story 3.3 state module:**
- Import `readSessionState()` from the session persistence module built in Story 3.3
- Do NOT re-implement file reading here — use the shared module to stay DRY

**Depends on Story 3.1 agent file parser:**
- When agent is active, load that agent's definition from `.aiox-core/development/agents/<agent-id>/`
- Use the parser from Story 3.1 to extract commands and scope
- If Story 3.1 parser is not yet available, implement inline and note for later refactor

**Context detection decision tree:**
```
1. If --raw flag present → render static help, exit
2. Try readSessionState()
   - On exception or null → render static help, exit
3. If session.activeAgent is set → show agent-specific commands and scope
4. Else if session.activeStoryId is set → show story-phase next actions
5. Else → show onboarding guidance (no active story)
```
Note: Agent check (step 3) takes priority over story check (step 4) because an active agent is the most specific context.

**Phase-to-next-action mapping (same as Story 3.3 suggestion logic — reuse if extracted to shared module):**
- `Draft` → Suggest @po validation
- `Ready` → Suggest starting development with @dev
- `InProgress` → Suggest continuing implementation
- `InReview` → Suggest checking QA status
- `Done` → Suggest creating next story with @sm

**Output format:**
- Use consistent section headers (same style as `aiox status` output)
- Context-specific output appears before the standard "Run `aiox help --raw` for full command reference" footer line
- Do NOT suppress the footer line in any branch — always show it

**Performance constraint (AC7):**
- The only I/O in context detection is `readSessionState()` which is a synchronous `fs.readFileSync` on a local file
- No network calls, no async operations in the detection path
- If the file does not exist, `fs.existsSync` check returns immediately

**Constitution Article IV (No Invention):**
- All suggested commands in context-aware output must be actual existing AIOX commands (verified against `bin/aiox.js` command registry)
- Do not suggest commands that do not exist in the CLI

**Constitution Article I (CLI First):**
- No UI component; pure terminal output
- Output must be readable with plain `cat` (no interactive mode in base `aiox help`)
