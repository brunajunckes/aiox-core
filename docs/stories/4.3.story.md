# Story 4.3: Community Feedback Loop

## Objective

Add a structured `aiox feedback` CLI command that collects NPS score + free-text feedback from users at key journey points (post-onboarding, post-first-story). Feedback is stored locally with zero external dependencies. Optionally posts to GitHub Discussions via `gh` CLI when the user explicitly requests it.

## Agent

@dev (Dex)

## Status

InProgress

## Story

**As a** developer using AIOX,
**I want** a simple way to submit structured feedback at key moments in my journey,
**so that** the AIOX team can understand user satisfaction and surface real pain points — with my feedback staying local unless I choose to share it.

## Acceptance Criteria

- [x] `aiox feedback` prompts for NPS score (1-10) and optional free-text comment, then saves locally
- [x] `aiox feedback submit` posts the most recent unsent local feedback to GitHub Discussions via `gh` CLI
- [x] `aiox feedback list` displays all locally stored feedback entries with timestamps
- [x] Feedback is stored in `.aiox/feedback/` as JSON files (one per entry: `{timestamp}.json`)
- [x] Feedback schema: `{ id, timestamp, nps: number, comment: string, trigger: string, submitted: boolean }`
- [x] Trigger points: `post-onboarding` (after quickstart completes), `post-first-story` (after first story is marked Done)
- [x] At trigger points, AIOX prompts the user with: "Got a minute? Share feedback on your experience [y/N]:" — non-blocking, defaults to No
- [x] `aiox feedback submit` requires `gh` CLI to be authenticated; if not, prints clear error with auth instructions
- [x] If `gh` CLI is absent, `aiox feedback submit` exits gracefully with install instructions — does not crash
- [x] Zero external npm dependencies for feedback collection (use Node.js built-ins + `gh` CLI as optional)
- [x] Unit tests cover: feedback schema validation, local storage, trigger prompt logic, graceful degradation when `gh` absent

## Dev Notes

**GitHub Discussions Target:**
The target repo and discussion category for `aiox feedback submit` should be configurable. For now, hardcode as a constant in the module — do not read from a remote config. Example:
```javascript
const FEEDBACK_REPO = 'synkra/aiox-core'; // owner/repo
const FEEDBACK_CATEGORY = 'Feedback';     // GitHub Discussion category name
```

**Non-Interactive Detection:**
Before showing interactive prompts (feedback collection, trigger hooks), check if stdin is a TTY:
```javascript
if (!process.stdin.isTTY) return; // skip prompts in CI / piped contexts
```

**Trigger Hook — First Story Only:**
The `post-first-story` trigger should fire only once. Track in `.aiox/feedback/state.json`: `{ firstStoryFeedbackSent: boolean }`. Check this flag before prompting.

**File Locations:**
- Feedback entries: `.aiox/feedback/{ISO-timestamp}.json`
- Trigger state: `.aiox/feedback/state.json`
- Module: consistent with other CLI modules from Stories 4.1 and 4.2
- Tests: `tests/feedback/` or existing test directory

**gh CLI GraphQL Mutation for GitHub Discussions:**
Use `gh api graphql` with the `createDiscussion` mutation. The repository ID and category ID must be resolved once via `gh api graphql` (can be hardcoded as constants after first lookup — document them in Dev Notes at implementation time).

**Atomic Writes:**
Same pattern as Story 4.1 — write to temp file, rename to final path.

**Testing:**
- Mock `child_process.execSync` or equivalent for `gh` CLI calls
- Test TTY detection by overriding `process.stdin.isTTY` in tests
- Test location: `tests/feedback/`
