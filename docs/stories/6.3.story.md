# Story 6.3: Plugin Hook System

## Objective

Implement a lightweight hook system that lets users register arbitrary shell commands to run at well-defined points in the AIOX CLI lifecycle (pre-command, post-command, pre-commit, post-test). Hooks are stored in `.aiox/hooks.json` and managed through `aiox hooks` subcommands, enabling extensibility without modifying framework core.

## Agent

@dev (Dex)

## Status

Draft

## Story

**As a** developer integrating AIOX into a custom workflow,
**I want** to register shell commands that run automatically at CLI lifecycle events,
**so that** I can extend AIOX behaviour (e.g. notify Slack after a command, run a linter before commit) without touching the framework source code.

## Acceptance Criteria

- [ ] `aiox hooks list` prints all registered hooks in a table: ID, Event, Command, Enabled
- [ ] `aiox hooks add <event> <command>` registers a hook and writes it to `.aiox/hooks.json`
- [ ] Supported events: `pre-command`, `post-command`, `pre-commit`, `post-test`
- [ ] `aiox hooks remove <id>` removes the hook with the given ID from `.aiox/hooks.json`
- [ ] `aiox hooks enable <id>` and `aiox hooks disable <id>` toggle a hook without removing it
- [ ] Hook IDs are auto-generated as short UUIDs (8 hex chars) on `add`
- [ ] Hooks registered for `pre-command` and `post-command` receive the subcommand name as `$AIOX_COMMAND` env var
- [ ] Hook execution uses `child_process.execSync`; a non-zero exit from a pre-* hook aborts the CLI command with exit code 1 and prints the hook's stderr
- [ ] Post-* hook failures are logged to stderr but do NOT abort the CLI command (fire-and-forget semantics)
- [ ] `.aiox/hooks.json` is created automatically if it does not exist; schema: `{ "hooks": [ { "id", "event", "command", "enabled" } ] }`
- [ ] Unit tests cover: add/remove/list/enable/disable operations, pre-hook abort behaviour, post-hook non-abort, env var injection

## Dev Notes

**Architecture Constraint — CLI First (Article I):**
Hooks extend the CLI lifecycle only. No UI hook management. `.aiox/hooks.json` is the single source of truth.

**Zero External Dependencies:**
Use only Node.js built-ins: `fs`, `path`, `crypto`, `child_process`. No new npm packages.

**hooks.json Schema:**
```json
{
  "hooks": [
    {
      "id": "a1b2c3d4",
      "event": "pre-command",
      "command": "echo $AIOX_COMMAND",
      "enabled": true
    }
  ]
}
```

**Executor Pattern:**
```javascript
function runPreHooks(event, env) {
  const hooks = loadHooks().filter(h => h.event === event && h.enabled);
  for (const hook of hooks) {
    try {
      execSync(hook.command, { stdio: ['ignore', 'inherit', 'pipe'], env, shell: true });
    } catch (err) {
      process.stderr.write(err.stderr || '');
      return { abort: true };
    }
  }
  return { abort: false };
}
```

**Security Note:**
The `command` field is stored and executed as a shell string. This is intentional — it is a user-defined automation tool. Document clearly that hooks run with the same privileges as the CLI process.

**Testing:**
- Test location: `tests/unit/hooks/`
- Mock `execSync` in unit tests to simulate hook exit codes without running real processes
- Mock `fs` for store read/write tests

**Relevant Source Tree:**
- CLI router: `bin/aiox.js`
- AIOX home dir resolution: check existing `config` or `status` commands for `AIOX_HOME` pattern
- Existing commands: `packages/cli/commands/` — follow same module structure
