# Story 6.3: Command Aliases & Shortcuts

## Objective

Implement a command alias system for the AIOX CLI. Users can define, list, and remove custom short-name aliases for any CLI command. Aliases are persisted in `.aiox/aliases.json` and resolved at command dispatch time. A set of built-in aliases ships with the framework by default. Zero external dependencies.

## Story

**As a** developer using AIOX daily,
**I want** to create short aliases for frequently used commands,
**so that** I can invoke complex or long commands with a single memorable short name without typing the full command string each time.

## Acceptance Criteria

- [ ] `aiox alias set <name> <command...>` creates or overwrites an alias named `<name>` that expands to `<command...>` (command may include subcommands and flags)
- [ ] `aiox alias list` prints a table of all defined aliases (built-in and user-defined) with columns: Name | Expands To | Source (built-in / user)
- [ ] `aiox alias remove <name>` removes a user-defined alias by name and prints confirmation on success
- [ ] `aiox alias remove <name>` fails with a clear error message if the alias does not exist or if the target is a built-in alias (built-ins cannot be removed)
- [ ] User-defined aliases are persisted to `.aiox/aliases.json`; the file is created automatically on first alias creation
- [ ] Built-in aliases are defined in source code (not in `.aiox/aliases.json`) and are always available: `t` → `telemetry status`, `h` → `health`, `s` → `smoke-test`
- [ ] When `aiox <name>` is invoked and `<name>` matches a registered alias, the CLI expands the alias and executes the resolved command
- [ ] Additional arguments passed after the alias name are appended to the expanded command: `aiox t --verbose` resolves to `aiox telemetry status --verbose`
- [ ] Alias names must be alphanumeric + hyphens only; `aiox alias set` rejects names containing spaces or special characters with a clear validation error
- [ ] Creating an alias with the same name as a built-in CLI command (e.g., `aiox alias set help something`) is blocked with a clear error: "Cannot shadow built-in command '{name}'"
- [ ] `aiox alias list` still shows built-in aliases when `.aiox/aliases.json` is empty or missing, with a note that no user aliases are defined
- [ ] Unit tests cover: alias set (create, overwrite, validation error), alias list (empty + populated), alias remove (success, not-found error, built-in error), alias resolution at dispatch, argument appending after alias expansion

## Dev Notes

**Architecture Constraint — CLI First (Article I):**
Aliases are a CLI-only construct. No dashboard, no UI for alias management. `aiox alias` commands are the canonical interface.

**Zero External Dependencies:**
Use only Node.js built-ins: `fs`, `path`. No new npm packages. Aliases file is plain JSON.

**Aliases JSON Shape:**
```javascript
// .aiox/aliases.json
{
  "deploy-dry": "smoke-test --bail",
  "ts": "telemetry status --format json"
}
```

**Built-in Aliases (source-defined, not in JSON):**
```javascript
const BUILTIN_ALIASES = {
  t: 'telemetry status',
  h: 'health',
  s: 'smoke-test',
};
```

**Resolution Pattern in Dispatcher:**
```javascript
function resolveAlias(args, allAliases, depth = 0) {
  if (depth > 10) throw new Error(`Circular alias detected: ${args[0]}`);
  const expansion = allAliases[args[0]];
  if (!expansion) return args;
  const expanded = expansion.split(' ');
  const resolved = [...expanded, ...args.slice(1)];
  return resolveAlias(resolved, allAliases, depth + 1);
}
```

**Argument Appending Example:**
```
User types:   aiox t --format json
alias t =     telemetry status
Resolved:     aiox telemetry status --format json
```

**Coordination with Story 6.1 (Plugin System):**
Both stories extend the CLI command registry. Ensure `resolveAlias` runs before plugin command dispatch so alias names can also expand to plugin commands — this works naturally since resolution happens at invocation time.

**Testing:**
- Test location: `tests/aliases/` following existing test directory convention
- Mock `fs` reads/writes in unit tests — do not touch actual `.aiox/aliases.json`
