# Story 6.4: Release Preparation & Version Bump

## Objective

Implement the `aiox release` command family to automate the v5.1.0 release workflow: pre-release checklist execution (`prepare`), semantic version bumping (`bump`), release notes generation from the changelog (`notes`), and a readiness check (`check`) that gates the release on clean state and passing tests. Integrates with the changelog module shipped in Story 5.2.

## Agent

@dev (Dex)

## Status

Draft

## Story

**As a** maintainer preparing an AIOX release,
**I want** a single `aiox release` command family that automates the release checklist,
**so that** I can produce a consistent, well-documented release without running manual steps or forgetting to bump the version.

## Acceptance Criteria

- [ ] `aiox release check` validates release readiness: exits 0 if all checks pass, 1 if any fail
- [ ] `aiox release check` verifies: (a) no uncommitted changes (`git status --porcelain` is empty), (b) all unit tests pass (`npm test`), (c) ESLint passes (`npm run lint`), (d) `CHANGELOG.md` has an entry for the current package version
- [ ] Each check in `release check` is reported individually as PASS or FAIL with a summary at the end
- [ ] `aiox release bump <major|minor|patch>` reads `package.json`, increments the specified version segment, writes the updated version back, and prints the old and new version
- [ ] `aiox release bump` validates that the argument is one of `major`, `minor`, `patch`; exits 1 with a helpful message otherwise
- [ ] `aiox release notes` reads `CHANGELOG.md` and extracts the section for the most recent version tag (everything between the first `## [x.y.z]` heading and the next such heading)
- [ ] `aiox release notes --since <version>` extracts the section for the specified version instead
- [ ] `aiox release prepare` runs the full sequence: `release check` → `release bump <type>` (prompts for type if not given via `--type`) → `release notes` → prints final summary
- [ ] `aiox release prepare --type patch` runs non-interactively (YOLO mode), using the given bump type
- [ ] All subcommands print clean, structured output; errors go to stderr; exit codes are strictly 0 (success) or 1 (failure)
- [ ] Unit tests cover: version parsing and bump logic, changelog section extraction, check result aggregation, exit code contract

## Dev Notes

**Architecture Constraint — CLI First (Article I):**
`aiox release` is the canonical release gate. No UI. Designed to run in CI as well as locally: all output is machine-parseable when piped, errors go to stderr.

**Zero External Dependencies:**
Use only Node.js built-ins: `fs`, `path`, `child_process`, `process`. No semver library — version strings are always `major.minor.patch`.

**Dependency — Story 5.2 (Changelog):**
The `release check` (Check D) and `release notes` commands read `CHANGELOG.md` produced by the changelog module. Confirm `CHANGELOG.md` path convention from Story 5.2 before implementing.

**Semver Bump (no library):**
```javascript
function bumpVersion(version, type) {
  const [major, minor, patch] = version.split('.').map(Number);
  if (type === 'major') return `${major + 1}.0.0`;
  if (type === 'minor') return `${major}.${minor + 1}.0`;
  if (type === 'patch') return `${major}.${minor}.${patch + 1}`;
  throw new Error(`Invalid bump type: ${type}`);
}
```

**Changelog Section Extraction:**
```javascript
const HEADING_RE = /^## \[(\d+\.\d+\.\d+)\]/gm;
function extractSection(content, version) {
  const parts = content.split(HEADING_RE);
  const idx = parts.indexOf(version);
  if (idx === -1) return null;
  return `## [${version}]${parts[idx + 1]}`;
}
```

**Atomic package.json Write:**
```javascript
const tmpPath = packageJsonPath + '.tmp';
fs.writeFileSync(tmpPath, JSON.stringify(pkg, null, 2) + '\n');
fs.renameSync(tmpPath, packageJsonPath);
```

**Process Spawn Pattern:**
Use `spawnSync` with `stdio: 'pipe'` for `npm test` and `npm run lint` so output is captured and only shown on failure. Pass `{ shell: true }` for cross-platform compatibility.

**Testing:**
- Test location: `tests/unit/release/`
- Mock `spawnSync` and `fs` in unit tests — do not run real `npm test` or write real `package.json`
- Use fixture strings for changelog content tests

**Relevant Source Tree:**
- `package.json` — version source of truth
- `CHANGELOG.md` — written by Story 5.2 changelog module
- CLI router: `bin/aiox.js`
- Smoke test runner (Story 5.1): `packages/cli/commands/smoke-test.js` — follow same subprocess pattern
