# Story 7.1: Workflow Automation Engine

## Objective

Introduce a workflow execution layer to the AIOX CLI so that developers can define and run named multi-step workflows composed of existing CLI commands. Running `aiox workflow run full-cycle` executes lint, tests, changelog generation, and a release check in sequence. Workflows are defined in `.aiox/workflows/*.yaml` and support conditions, parallel steps, and custom sequences.

@dev (Dex)

## Status

## Story

**As a** developer running repetitive multi-step CLI sequences,
**I want** to define named workflows in YAML and execute them with a single command,
**so that** I can automate my development pipeline without writing shell scripts and keep workflows version-controlled alongside the project.

## Acceptance Criteria

- [ ] `aiox workflow run <name>` executes a workflow defined in `.aiox/workflows/<name>.yaml`
- [ ] `aiox workflow list` lists all available workflows found in `.aiox/workflows/`
- [ ] `aiox workflow create <name>` scaffolds a new workflow YAML template at `.aiox/workflows/<name>.yaml`
- [ ] Workflow YAML supports `steps` array where each step references any aiox CLI command via `run:` field
- [ ] Steps execute sequentially by default; steps marked `parallel: true` run concurrently
- [ ] A step with `condition: <expr>` is skipped when the expression evaluates to false (e.g. `condition: "{{ env.CI }} == 'true'"`)
- [ ] Built-in workflow `full-cycle` is bundled: runs lint → test → changelog → release check in sequence
- [ ] Built-in workflow `quick-check` is bundled: runs lint → test in sequence
- [ ] On step failure the run halts and prints which step failed and its exit code; `--continue-on-error` flag skips failures
- [ ] `aiox workflow run <name> --dry-run` prints the resolved execution plan without executing any steps
- [ ] Zero external npm dependencies — implemented with Node.js built-ins and the existing aiox CLI runner

## Dev Notes

**Architecture Constraint — CLI First (Article I):**
All workflow logic is CLI-only. Workflows are defined in `.aiox/workflows/` — a project-level directory within the framework boundary (L4, always mutable).

**YAML Parsing Without External Deps:**
Node.js has no built-in YAML parser. Use a minimal hand-rolled parser for the simple workflow schema (keys, string values, booleans), OR check if the project already vendors a YAML parser in `packages/`. If neither is available, represent the workflow format in JSON (`.aiox/workflows/*.json`) for Sprint 7 and add a YAML note in Dev Notes for future sprint. Document this decision as `[AUTO-DECISION]`.

[AUTO-DECISION] YAML vs JSON format → YAML preferred for readability; implement a minimal key-value YAML parser covering only the workflow schema subset (strings, booleans, arrays of objects). No `js-yaml` dependency. (reason: Zero-external-deps constraint from AC-11; full YAML spec not needed for this schema)

**Child Process Execution:**
Use `child_process.spawnSync` for sequential steps (simpler, blocking). Use `child_process.spawn` with Promise wrappers for parallel groups. Inherit stdio so step output streams directly to terminal.

**Condition Expression Format:**
Keep it simple: support only `{{ env.VAR_NAME }} == 'literal'` and `{{ env.VAR_NAME }} != 'literal'`. No arithmetic or logical operators in Sprint 7. Parse with a regex, resolve `process.env[VAR_NAME]`, compare.

**Relevant Source Tree:**
- CLI router: `bin/aiox.js` (subcommand dispatch)
- Existing command modules: `packages/cli/commands/`
- Workflow storage: `.aiox/workflows/` (create if absent)
- Entity registry: `.aiox-core/data/entity-registry.yaml`

**Testing:**

- Test location: `tests/unit/workflow/`
- Use Jest with `jest.spyOn(childProcess, 'spawnSync')` to mock process execution
- Do not spawn real child processes in unit tests
- Test the loader separately from the executor (pure function tests for loader)
- Integration test: run `aiox workflow run quick-check --dry-run` against a fixture workflow file
