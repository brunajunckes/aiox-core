# Story 6.1: Plugin System Architecture

## Objective

Implement a plugin system that allows third-party developers to extend the AIOX CLI with new subcommands. Running `aiox plugin init` scaffolds a plugin directory, `aiox plugin register <path>` loads it into the CLI, and `aiox plugin list` shows all registered plugins. Plugins export a `commands` array that the CLI dispatcher resolves at startup — zero external dependencies, pure Node.js.

## Story

**As a** developer building on top of AIOX,
**I want** a plugin system that lets me register custom CLI commands without modifying the core framework,
**so that** I can extend AIOX for project-specific needs while keeping my extensions isolated and portable.

## Acceptance Criteria

- [ ] `aiox plugin init` creates a new plugin directory with `aiox-plugin.json` manifest and a starter `index.js` at the specified path (or current directory if path omitted)
- [ ] `aiox-plugin.json` manifest contains required fields: `name` (string), `version` (string), `description` (string), `commands` (array of command descriptor objects)
- [ ] Each command descriptor in `commands` has: `name` (string), `description` (string), `handler` (string — relative path to handler file)
- [ ] `aiox plugin register <path>` validates the manifest at `<path>/aiox-plugin.json`, adds the resolved absolute path to `.aiox/plugins.json`, and prints confirmation
- [ ] `aiox plugin register` fails with a clear error message if: manifest file is missing, manifest JSON is invalid, required fields are absent, or plugin with the same name is already registered
- [ ] `aiox plugin list` reads `.aiox/plugins.json` and prints a table of registered plugins: name, version, path, and command count
- [ ] `aiox plugin list` prints "No plugins registered." when `.aiox/plugins.json` is empty or does not exist
- [ ] On CLI startup, the dispatcher loads all registered plugins from `.aiox/plugins.json`, imports each plugin's `index.js`, and merges exported `commands` into the command registry
- [ ] A plugin command is invocable as `aiox <plugin-command-name>` using the same routing as built-in commands
- [ ] Plugin load errors (missing file, bad export, runtime exception) are caught, printed as warnings, and do not prevent the CLI from starting
- [ ] Zero new external npm dependencies introduced — uses only Node.js built-ins (`fs`, `path`, `module`)
- [ ] Unit tests cover: manifest validation, plugin registration (success + error paths), plugin list output, CLI startup plugin loading, error isolation on bad plugin

## Dev Notes

**Architecture Constraint — CLI First (Article I):**
The plugin system extends the CLI exclusively. No UI, no config dashboard. Plugins are registered and invoked purely via CLI commands.

**Zero External Dependencies:**
Use only Node.js built-ins: `fs`, `path`, `module`. No new npm packages. Plugin manifests are plain JSON, no YAML, no schema validation libraries.

**Plugin Registry File Shape:**
```javascript
// .aiox/plugins.json
[
  {
    "name": "my-plugin",
    "version": "1.0.0",
    "path": "/absolute/path/to/plugin",
    "registeredAt": "2026-04-09T00:00:00.000Z"
  }
]
```

**Plugin index.js Contract:**
```javascript
// {plugin-dir}/index.js
module.exports = {
  commands: [
    {
      name: 'my-cmd',
      description: 'Does something custom',
      handler: async (args) => {
        // implementation
      },
    },
  ],
};
```

**Startup Loader Pattern:**
```javascript
function loadPlugins(registry) {
  for (const entry of registry) {
    try {
      const plugin = require(path.join(entry.path, 'index.js'));
      if (Array.isArray(plugin.commands)) {
        for (const cmd of plugin.commands) {
          commandRegistry.register(cmd.name, cmd);
        }
      }
    } catch (err) {
      console.warn(`[WARN] Plugin "${entry.name}" failed to load: ${err.message}`);
    }
  }
}
```

**Boilerplate `aiox-plugin.json` generated by `plugin init`:**
```json
{
  "name": "my-plugin",
  "version": "1.0.0",
  "description": "My AIOX plugin",
  "commands": [
    {
      "name": "my-cmd",
      "description": "Example command",
      "handler": "./commands/my-cmd.js"
    }
  ]
}
```

**Testing:**
- Test location: `tests/plugins/` following existing test directory convention
- Mock `require()` for plugin loader tests using Jest's module mocking
- Do not load actual plugins from disk in unit tests
