# Story 9.3: Error Catalog & Diagnostic Messages

## Objective

Create a centralized error catalog that maps error codes to human-readable messages with fix suggestions, replacing ad-hoc error strings across the CLI with consistent, actionable diagnostics that help users self-resolve issues without external documentation.

## Agent

@dev (Dex)

## Status

InReview

## Story

**As a** developer using the AIOX CLI,
**I want** consistent error messages with error codes and fix suggestions,
**so that** I can quickly understand and resolve issues without searching documentation.

## Acceptance Criteria

- [x] All CLI errors include an error code in format `AIOX-ENNN` (e.g., AIOX-E001)
- [x] Error catalog is stored in `.aiox-core/data/error-catalog.yaml` with fields: code, message, suggestion, severity
- [x] `aiox doctor --errors` lists all known error codes with descriptions
- [x] `aiox explain AIOX-E001` shows the full error details, suggestions, and related documentation
- [x] CLI commands use `throwAioxError(code, context)` instead of raw `throw new Error()`
- [x] Error messages include: error code, description, suggestion, and `Run 'aiox explain AIOX-ENNN' for details`
- [x] At least 20 common errors are cataloged (install failures, config issues, story problems, git errors)
- [x] Zero external npm dependencies

- Interactive error recovery wizards (future enhancement)
- Localization of error messages (future enhancement)
- Real-time error aggregation dashboard (future enhancement)
- Migration of all existing errors (only top-5 common paths in this sprint)

## Dev Notes

**Architecture Constraint — CLI First (Article I):**
Error catalog is a read-only data structure that supports CLI observability. The catalog lives in `.aiox-core/data/`, is loaded on-demand, and cached in memory. No external APIs or network calls.

**Error Code Allocation Scheme:**
```
AIOX-ENNN where NNN = 001-999
Category prefix (in name): INSTALL, CONFIG, STORY, GIT, FRAMEWORK, RUNTIME
Example: AIOX-E001 (INSTALL: Missing node_modules)
```

**Error Catalog YAML Structure:**
```yaml
errors:
  - code: AIOX-E001
    category: INSTALL
    severity: critical
    message: "Failed to install AIOX: missing Node.js or npm"
    suggestion: "Install Node.js 18+ from https://nodejs.org and retry 'npx aiox-core install'"
    relatedDocs: "docs/setup.md"
```

**Context Interpolation:**
Error messages support `${variable}` syntax for dynamic values:
```javascript
throwAioxError('AIOX-E042', { storyId: '9.3', expected: 'Draft', actual: 'InProgress' })
// Fills: "Story ${storyId} is ${actual}, expected ${expected}"
```

**Graceful Fallback:**
If error catalog fails to load, CLI uses a minimal fallback error and logs warning:
```javascript
if (!catalog) {
  console.warn('[WARN] Error catalog unavailable; using minimal error output')
  throw new Error(`${code}: ${errorMessage}`)
}
```

**Relevant Source Tree:**
- CLI router: `bin/aiox.js`
- Utility modules: `.aiox-core/cli/utils/error.js`, `.aiox-core/cli/utils/error-catalog.js`
- Error catalog: `.aiox-core/data/error-catalog.yaml`
- Commands: `.aiox-core/cli/commands/explain.js`, `.aiox-core/cli/commands/doctor.js` (extended)
- Entity registry: `.aiox-core/data/entity-registry.yaml`

**Testing:**

- Use fixture YAML files in test directory for validation tests
- Test error message formatting with string.includes() assertions
- Test fuzzy matching with predefined test vectors
- Do not test actual CLI execution; test the modules independently and the commands with mocked dependencies

**Top-5 Error Paths to Retrofit:**
1. AIOX-E042: Story status validation (e.g., Cannot set story status to invalid value)
2. AIOX-E103: Configuration file parse error (e.g., core-config.yaml malformed)
3. AIOX-E051: Git operation failure (e.g., git push auth failure)
4. AIOX-E022: Template loading error (e.g., template file not found)
5. AIOX-E081: Task execution failure (e.g., task script returned non-zero exit code)
