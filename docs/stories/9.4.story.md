# Story 9.4: CLI Command Aliases & Shortcuts

## Objective

Allow developers to define custom command aliases (short names for frequently used commands) via a simple YAML config, reducing keystrokes for power users while keeping the full command set available for discoverability.

## Agent

@dev (Dex)

## Status

## Story

**As a** power user of the AIOX CLI,
**I want** to define short aliases for frequently used commands,
**so that** I can work faster without typing long command names.

## Acceptance Criteria

- [x] `aiox alias set <name> <command>` creates a new alias (e.g., `aiox alias set s status`)
- [x] `aiox alias list` shows all defined aliases in a table format
- [x] `aiox alias remove <name>` deletes an alias
- [x] Aliases are stored in `.aiox/aliases.yaml` as `{ name: command }` pairs
- [x] Running `aiox <alias>` executes the mapped command with any additional args forwarded
- [x] Built-in aliases ship by default: `s` → `status`, `d` → `doctor`, `h` → `help`, `a` → `agents`
- [x] `aiox alias reset` restores default aliases
- [x] Alias names cannot conflict with existing command names (exits with error)
- [x] `aiox alias set` without args shows help text
- [x] Zero external npm dependencies

## Risks

1. **Alias → Command Conflict:** User creates alias that shadows built-in command
   - **Mitigation:** Validation rejects conflicting names with clear error message

2. **Alias Forwarding Edge Case:** Args containing spaces or special characters not forwarded correctly
   - **Mitigation:** Use shell-safe arg parsing, comprehensive unit tests

3. **YAML Corruption:** Manual editing of `.aiox/aliases.yaml` breaks file format
   - **Mitigation:** Reset command restores defaults; validation on load

## Dev Notes

**Architecture Constraint — CLI First (Article I):**
All alias operations go through the CLI. The alias system is a pure CLI feature with no UI component. Resolution happens in the CLI router before command dispatch.

**Test Location:** `tests/cli/alias.test.js`

**Module Organization:**
```
.aiox-core/cli/commands/alias/
├── index.js            # CRUD handlers (set, list, remove, reset, help)
├── store.js            # Persistence (load, save, validate)
└── __tests__/
    └── alias.test.js   # Unit tests
```

**Critical Patterns:**

```javascript
// store.js — Load/Save Aliases
function loadAliases() {
  const aliasFile = path.join(process.env.AIOX_HOME || '.aiox', 'aliases.yaml');
  if (!fs.existsSync(aliasFile)) {
    return getDefaultAliases();
  }
  try {
    const yaml = require('yaml'); // Built-in or minimal YAML parser
    const content = fs.readFileSync(aliasFile, 'utf8');
    return yaml.parse(content) || getDefaultAliases();
  } catch (err) {
    console.error(`Failed to load aliases: ${err.message}, using defaults`);
    return getDefaultAliases();
  }
}

function saveAliases(aliases) {
  const yaml = require('yaml');
  const aliasFile = path.join(process.env.AIOX_HOME || '.aiox', 'aliases.yaml');
  const content = yaml.stringify(aliases);
  fs.writeFileSync(aliasFile, content, 'utf8');
}

// router.js — Resolve Aliases
function resolveAlias(input, aliases) {
  if (aliases[input]) {
    return { isAlias: true, command: aliases[input] };
  }
  return { isAlias: false, command: input };
}

// In main CLI router:
async function routeCommand(args) {
  const aliases = loadAliases();
  const [command, ...rest] = args;
  const resolved = resolveAlias(command, aliases);
  
  if (resolved.isAlias) {
    return await executeCommand(resolved.command, rest);
  }
  return await executeCommand(command, rest);
}
```

**Relevant Source Tree:**
- CLI entry point: `bin/aiox.js`
- Command router: `packages/cli/router.js`
- Command registry: `packages/cli/commands/` (reference for existing commands)
- Help system: `packages/cli/help.js`
- Config/state dir: `.aiox/`

**YAML Library Decision:**
- Prefer lightweight YAML parser (e.g., `js-yaml` if already in deps, or simple hand-rolled for minimal overhead)
- NO new npm dependencies — use built-in if possible, or simple inline YAML serialization

**Testing Philosophy:**
- No external dependencies for alias module
- Mock file system for unit tests (use Node.js `fs` module directly)
- Test both happy path and error cases
- Verify conflict detection blocks invalid aliases
- Verify arg forwarding works with spaces and special chars

**Edge Cases:**
1. User creates alias that matches command name → ERROR with suggestion
2. User tries to alias to non-existent command → Allow (might be future command)
3. Alias contains spaces in name → REJECT with error message
4. Alias value is empty string → REJECT with error message
5. `.aiox/aliases.yaml` corrupted → Load defaults, log warning
6. Recursive alias (a → b, b → a) → Resolve max depth 1, then treat as unknown command
