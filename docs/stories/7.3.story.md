# Story 7.3: Code Stats & Complexity Dashboard

## Objective

Add a `aiox stats` command that gives developers an instant snapshot of codebase health: lines of code, file count, language breakdown, cyclomatic complexity per module, and historical trend over recent commits. All analysis reads source files directly using Node.js built-ins — no external dependencies.

## Agent

@dev (Dex)

## Status

Draft

## Story

**As a** developer or tech lead monitoring codebase growth,
**I want** a CLI command that shows LOC, file counts, language distribution, complexity scores, and commit trends,
**so that** I can detect scope creep, identify complex hotspots, and track codebase health over time without leaving the terminal.

## Acceptance Criteria

- [ ] `aiox stats` prints a codebase summary: total LOC, blank lines, comment lines, file count, and language breakdown table
- [ ] `aiox stats --complexity` adds a cyclomatic complexity score per JavaScript/TypeScript module, sorted descending by score
- [ ] `aiox stats --history` shows a trend table of LOC and file count for the last 10 commits (uses `git log` + `git show`)
- [ ] `aiox stats --history --limit <n>` accepts a custom commit count (default 10, max 50)
- [ ] `aiox stats --json` outputs all findings as a single JSON object for machine consumption
- [ ] Language detection is based on file extension mapping (`.js` → JavaScript, `.ts` → TypeScript, `.py` → Python, `.md` → Markdown, `.yaml`/`.yml` → YAML, `.json` → JSON, etc.)
- [ ] Complexity analysis covers only JS/TS files; complexity score = number of decision points (if, else if, for, while, switch case, catch, ternary `?`, `&&`, `||`) + 1
- [ ] Files in `node_modules/`, `.git/`, and `dist/` are excluded from all scans
- [ ] Zero external npm dependencies — implemented with Node.js `fs`, `path`, `child_process` built-ins only
- [ ] Unit tests cover: LOC counter, language detector, complexity scorer, git log parser

  - [ ] Return `{ total, blank, comment, code }` per file; aggregate for summary

- [ ] Task 3: Language detector (AC: 1, 6, 9)

  - [ ] Extension map: `.js` → JavaScript, `.ts`/`.tsx` → TypeScript, `.py` → Python, `.md` → Markdown, `.yaml`/`.yml` → YAML, `.json` → JSON, `.sh` → Shell, `.css` → CSS, `.html` → HTML
  - [ ] Unknown extensions grouped under "Other"

- [ ] Task 4: Complexity scorer (AC: 2, 7, 9)
  - [ ] Create `packages/cli/stats/complexity-scorer.js` — reads JS/TS files, counts decision points
  - [ ] Decision point regex patterns: `\bif\b`, `\belse if\b`, `\bfor\b`, `\bwhile\b`, `\bcase\b`, `\bcatch\b`, ternary `?` (not inside strings), `&&`, `||`
  - [ ] Score = decision_points + 1
  - [ ] Return array of `{ filePath, score }` sorted descending
  - [ ] Mark files with score >= 10 as HIGH complexity, 5-9 as MEDIUM, < 5 as LOW

- [ ] Task 5: History analyzer (AC: 3, 4, 9)

## Dev Notes

**Architecture Constraint — CLI First (Article I):**
Stats is purely a CLI feature. No UI surface. Output goes to stdout; `--json` enables downstream piping.

**Cyclomatic Complexity Approximation:**
The implementation uses a heuristic regex count, not a proper AST parse. This is intentional (zero deps). Document in `aiox stats --complexity` output header: "Complexity scores are approximations based on decision-point counting, not full AST analysis."

**Comment Detection Heuristic:**
Line-level detection only (no block comment state machine needed for Sprint 7). Lines where `trimmed.startsWith('//')`, `trimmed.startsWith('#')`, `trimmed.startsWith('*')`, `trimmed.startsWith('/*')`, `trimmed.startsWith('*/')` are counted as comment lines.

**Performance Consideration:**
For the VPS environment (4C/16GB) with typical project sizes (< 10K files), synchronous `fs.readdirSync` + `fs.readFileSync` is acceptable. If a project exceeds 50K files, the traverser will be slow — document this as a known limitation.

**git Availability:**
Assume `git` is on PATH (it is present in all AIOX development environments). Use `child_process.execSync('git log ...', { encoding: 'utf8' })`. If `git` is not available or the directory is not a git repo, `--history` should print a clear error and exit 0 (not crash).

**Relevant Source Tree:**
- CLI router: `bin/aiox.js`
- Existing command modules: `packages/cli/commands/`
- Source to scan: project root (respecting exclusions)
- Git integration: via `child_process` subprocess calls

**Testing:**

- Test location: `tests/unit/stats/`
- Use `tmp` directory helpers in Jest (`beforeAll`/`afterAll`) for traverser and LOC counter tests
- Mock `child_process.execSync` for all git-dependent tests
- Fixture JS files with known complexity: simple function (score 1), function with 3 ifs (score 4)
- Do not rely on the actual project's LOC counts in tests — use fixtures
