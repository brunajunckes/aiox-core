# Story 7.4: Agent Performance Benchmarks

## Objective

Add a `aiox benchmark` command that measures the execution time of every AIOX CLI command and produces a structured performance report. A `--compare` flag allows comparing timings against a saved baseline from a previous git ref, making it straightforward to detect performance regressions before they reach production.

## Agent

@dev (Dex)

## Status

Draft

## Story

**As a** developer maintaining the AIOX CLI,
**I want** a command that benchmarks each CLI subcommand and compares timings against a baseline,
**so that** I can detect performance regressions early and keep the CLI fast for all users.

## Acceptance Criteria

- [ ] `aiox benchmark` runs each top-level CLI subcommand with a `--help` or no-op invocation and records wall-clock execution time
- [ ] Benchmark output table shows: command name, average time (ms), p95 time (ms), min time (ms), max time (ms), and pass/fail status against a threshold
- [ ] Each command is sampled at least 3 times per benchmark run; the number of samples is configurable via `--samples <n>` (default 3, max 20)
- [ ] `aiox benchmark --compare <git-ref>` loads the baseline saved at that ref (from `.aiox/benchmarks/<git-ref>.json`) and prints a delta table showing time change per command (absolute ms and %)
- [ ] `aiox benchmark --save` saves the current run results to `.aiox/benchmarks/<current-git-sha>.json`
- [ ] A command is flagged as SLOW when its average time exceeds 500ms; threshold is configurable in `.aiox-core/data/benchmark-config.yaml`
- [ ] `aiox benchmark --json` outputs the full results object as JSON to stdout
- [ ] The benchmark runner gracefully handles commands that exit with non-zero codes (marks timing as collected, notes non-zero exit separately)
- [ ] Zero external npm dependencies — implemented with Node.js `child_process`, `fs`, `path`, `perf_hooks` built-ins
- [ ] Unit tests cover: timing aggregation (avg, p95, min, max), threshold evaluation, delta calculation, result serialization

## Dev Notes

**Architecture Constraint — CLI First (Article I):**
Benchmark is a CLI-only observability tool. Results stored in `.aiox/benchmarks/` (L4 — project runtime, always mutable). Config stored in `.aiox-core/data/` (L3 — project config, mutable with care).

**Timing Precision:**
Use `perf_hooks.performance.now()` for sub-millisecond precision. Pattern:
```js
const { performance } = require('perf_hooks');
const t0 = performance.now();
spawnSync('node', ['bin/aiox.js', cmd, '--help'], { encoding: 'utf8' });
const elapsed = performance.now() - t0;
```

**Command Invocation Strategy:**
Invoke commands as `node bin/aiox.js <command> --help` via `spawnSync`. This measures true cold-start CLI latency including module load time — the metric most relevant for developer experience. Do not `require()` command modules directly (that would skip module loading time).

**git SHA Resolution:**
For `--save`, get current SHA with:
```js
execSync('git rev-parse --short HEAD', { encoding: 'utf8' }).trim()
```
If not in a git repo, use timestamp as fallback filename.

For `--compare <ref>`, resolve ref to SHA:
```js
execSync(`git rev-parse --short ${ref}`, { encoding: 'utf8' }).trim()
```
Then look for `.aiox/benchmarks/<sha>.json`. If not found, also try `.aiox/benchmarks/<ref>.json` as a fallback (allows saving with tag names like `v1.0`).

**p95 Calculation with Small Samples:**
With the default of 3 samples, `Math.floor(3 * 0.95) = 2` — so p95 is the 3rd element (index 2) of the sorted array, which equals max. This is correct behavior for small samples and should be documented in the help text.

**Benchmark Storage Directory:**
Create `.aiox/benchmarks/` if it doesn't exist before writing. Add `.aiox/benchmarks/*.json` to `.gitignore` by default — benchmark results are local artifacts. Document this behavior in `--save` help text.

**Relevant Source Tree:**
- CLI router: `bin/aiox.js`
- Existing command modules: `packages/cli/commands/`
- Benchmark storage: `.aiox/benchmarks/` (create on first `--save`)
- Threshold config: `.aiox-core/data/benchmark-config.yaml`

**Testing:**

- Test location: `tests/unit/benchmark/`
- Mock all `child_process.spawnSync` and `child_process.execSync` calls
- Test pure functions (stats aggregator, delta calculator, threshold evaluator) without any mocking
- Use `jest.spyOn(fs, 'writeFileSync')` for serializer write tests to avoid filesystem side effects
- Fixture baseline JSON objects for delta calculator tests — do not depend on `.aiox/benchmarks/` directory existence
