# Story 10.3: Config Diff & Migration Tool

## Objective

Provide an `aiox config-diff` command that shows differences between current config and defaults, and an `aiox config-diff migrate` subcommand that migrates old config key formats to the current schema.

## Agent

@dev (Dex)

## Status

InReview

## Story

**As a** developer managing AIOX configuration,
**I want** to see how my config differs from defaults and migrate old formats,
**so that** I can keep my configuration current and understand customizations.

## Acceptance Criteria

- [x] `aiox config-diff` shows differences between current config and defaults
- [x] `aiox config-diff --format json` outputs diff as JSON
- [x] `aiox config-diff migrate` migrates old config keys to current schema
- [x] `aiox config-diff migrate --dry-run` previews migration without writing
- [x] Uses builtin defaults when no defaults file exists
- [x] Supports YAML config files (js-yaml with fallback parser)
- [x] Diff shows modified, added, and removed keys with summary
- [x] Migration handles known old key mappings (e.g., frameworkProtection -> boundary.frameworkProtection)
- [x] Handles missing config files gracefully
- [x] Command registered in bin/aiox.js

## Tasks

- [x] Create command module at `.aiox-core/cli/commands/config-diff/index.js`
- [x] Implement simple YAML parser (fallback for when js-yaml unavailable)
- [x] Implement flatten/unflatten utilities for dot-notation keys
- [x] Implement diff computation and formatting (text and JSON)
- [x] Implement migration rules and config writer
- [x] Register command in `bin/aiox.js`
- [x] Write tests with 30 test cases

## Dev Notes

- Standalone command `config-diff` to avoid conflicts with existing `config` command
- Simple YAML parser handles flat/shallow configs without js-yaml dependency
- 6 migration rules map old flat keys to new nested keys
- `serializeYaml` used as fallback when js-yaml not available for writing
